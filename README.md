# Завдання 1: Чому HTTP недостатньо? Дослідження обмежень

## Скільки HTTP-запитів робить WebSocket для встановлення з'єднання?

> **Для встановлення WebSocket-з'єднання виконується один HTTP-запит (Handshake), після чого з'єднання перемикається на протокол WebSocket.**

---

## Якщо в чаті 1000 користувачів і всі «поллять» кожну секунду — скільки запитів за хвилину отримує сервер? А при WebSocket?

> **При HTTP Polling сервер отримує 60 000 запитів за хвилину. При WebSocket — браузер питає один раз, після чого з'єднання відкрите і сервер сам надсилає дані коли вони змінюються.**

---

## Назвіть 3 реальних застосування WebSocket (чат, ..., ...).

> **Месенджери та чати, Фінансові та аналітичні платформи, Багатокористувацькі онлайн-ігри**

# Завдання 2: WebSocket Handshake — як HTTP перетворюється на WebSocket

## Ключові заголовки Handshake

| Заголовок | Хто надсилає | Призначення |
|------------|------------|------------|
| `Upgrade: websocket` | Клієнт | Ініціює (клієнт) та підтверджує (сервер) зміну протоколу на WebSocket |
| `Connection: Upgrade` | Клієнт | Вказує, що поточне HTTP-з'єднання має бути трансформоване (оновлене) |
| `Sec-WebSocket-Key` | Клієнт | Передає випадковий Base64-ключ для захисту від помилкових підключень |
| `Sec-WebSocket-Accept` | Сервер | Підтверджує отримання ключа через SHA-1 хеш із додаванням GUID-константи |
| `Sec-WebSocket-Version` | Клієнт | Узгодження версії WebSocket між браузером та сервером |
| `101 Switching Protocols` | Сервер | HTTP-код відповіді, що засвідчує успішний перехід на новий протокол |

---

## Навіщо потрібен Sec-WebSocket-Key? Від чого він захищає?

> **Головна причина існування Sec-WebSocket-Key — це захист від застарілих або неправильно налаштованих мережевих посередників (проксі-серверів, кешуючих серверів, фаєрволів)**

---

## Що означає код відповіді 101? Чим він відрізняється від 200?

> **Код відповіді 101 (Switching Protocols) означає, що сервер погодився змінити протокол зв'язку з HTTP на WebSocket. Після цього HTTP більше не використовується для обміну даними, а з'єднання переходить у режим WebSocket.**
>
> **Код відповіді 200 (OK) означає, що HTTP-запит успішно оброблено і сервер повернув результат. Після завершення обміну даними з'єднання може бути закрито або використано для наступних HTTP-запитів, але протокол не змінюється.**
>
> **Головна різниця полягає в тому, що код 101 переводить з'єднання на інший протокол і відкриває постійний двонаправлений канал зв'язку (Full-Duplex), тоді як код 200 підтверджує успішне виконання звичайного HTTP-запиту без зміни протоколу.**

---

## Після успішного Handshake — HTTP протокол продовжує працювати чи «замінюється»?

> **Після успішного Handshake HTTP-протокол повністю припиняє роботу і замінюється на WebSocket, оскільки той самий TCP-канал перемикається на принципово інший, двонаправлений формат передачі даних.**

# Завдання 3: WebSocket фрейми — структура повідомлень

## Структура WebSocket фрейму (спрощено)

```text
┌─────────────────────────────────────────────────────┐
│ FIN │ RSV1 │ RSV2 │ RSV3 │      Opcode (4 біти)     │
│  1  │  0   │  0   │  0   │ Text=1, Binary=2, Close=8│
├─────────────────────────────────────────────────────┤
│ MASK │            Payload Length                    │
│  1   │ 7 бітів (або 16/64 для великих даних)        │
├─────────────────────────────────────────────────────┤
│     Masking Key (4 байти, клієнт → сервер)          │
├─────────────────────────────────────────────────────┤
│                   Payload Data                      │
└─────────────────────────────────────────────────────┘
```

---

## Таблиця Opcode

| Opcode | Назва | Призначення |
|---------|---------|-------------|
| `0x1` | **Text Frame** | Передача текстових даних у форматі UTF-8 |
| `0x2` | **Binary Frame** | Передача бінарних даних (файли, зображення, аудіо тощо) |
| `0x8` | **Close Frame** | Коректне завершення WebSocket-з'єднання |
| `0x9` | **Ping Frame** | Перевірка доступності іншої сторони та підтримання з'єднання |
| `0xA` | **Pong Frame** | Відповідь на `Ping Frame` |

---

## Навіщо клієнт маскує (XOR) дані перед відправкою серверу?

> **Клієнт маскує дані за допомогою операції XOR для захисту від неправильно налаштованих або застарілих мережевих посередників (проксі, кешів, фаєрволів), які можуть помилково інтерпретувати WebSocket-трафік як звичайний HTTP-трафік. Маскування не є шифруванням і не забезпечує конфіденційність даних.**

---

## Чому сервер не маскує дані, що відправляє клієнту?

> **Сервер не маскує дані, оскільки основна загроза пов'язана саме з трафіком від клієнта до сервера. Відсутність маскування з боку сервера зменшує навантаження на процесор і спрощує обробку повідомлень, не створюючи додаткових ризиків.**

---

## Що станеться, якщо сервер отримає Close Frame — він повинен закрити з'єднання одразу чи спочатку відправити у відповідь теж Close Frame?

> **Якщо сервер отримує Close Frame, він повинен відправити у відповідь власний Close Frame для підтвердження завершення з'єднання. Лише після цього обидві сторони закривають TCP-з'єднання. Це забезпечує коректне та узгоджене завершення WebSocket-сесії.**
