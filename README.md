## Завдання 1: Чому HTTP недостатньо? Дослідження обмежень

### Скільки HTTP-запитів робить WebSocket для встановлення з'єднання?
* **WebSocket робить тільки один HTTP запит щоб встановити з'єднання**

### Якщо в чаті 1000 користувачів і всі «поллять» кожну секунду — скільки запитів за хвилину отримує сервер? А при WebSocket?
* **При HTTP Polling сервер отримує 60 000 запитів за хвилину. При WebSocket — браузер питає один раз, після чого з'єднання відкрите і сервер сам надсилає дані коли вони змінюються.**

### Назвіть 3 реальних застосування WebSocket (чат, ..., ...).

## Завдання 2: WebSocket Handshake — як HTTP перетворюється на WebSocket

### Ключові заголовки Handshake
| Заголовок                 | Хто надсилає | Призначення                                                              |
| ------------------------- | ------------ | ------------------------------------------------------------------------ |
| `Upgrade: websocket`      | Клієнт       | Ініціює (клієнт) та підтверджує (сервер) зміну протоколу на WebSocket    |
| `Connection: Upgrade`     | Клієнт       | Вказує, що поточне HTTP-з'єднання має бути трансформоване (оновлене)     |
| `Sec-WebSocket-Key`       | Клієнт       | Передає випадковий Base64-ключ для захисту від помилкових підключень     |
| `Sec-WebSocket-Accept`    | Сервер       | Підтверджує отримання ключа через SHA-1 хеш із додаванням GUID-константи |
| `Sec-WebSocket-Version`   | Клієнт       | Узгодження версії WebSocket між браузером та сервером                    |
| `101 Switching Protocols` | Сервер       | HTTP-код відповіді, що засвідчує успішний перехід на новий протокол      |

### Навіщо потрібен Sec-WebSocket-Key? Від чого він захищає?
* **Головна причина існування Sec-WebSocket-Key — це захист від застарілих або неправильно налаштованих мережевих посередників (проксі-серверів, кешуючих серверів, фаєрволів)**

## Завдання 3: WebSocket фрейми — структура повідомлень

### Структура WebSocket фрейму (спрощено):
```
┌─────────────────────────────────────────────────────┐
│ FIN │ RSV1 │ RSV2 │ RSV3 │      Opcode (4 біти)     │
│  1  │  0   │  0   │  0   │ Text=1, Binary=2, Close=8│
├─────────────────────────────────────────────────────┤
│ MASK │            Payload Length                    │
│  1   │ 7 бітів (або 16/64 для великих даних)        │
├─────────────────────────────────────────────────────┤
│     Masking Key (4 байти, клієнт → сервер)          │
├─────────────────────────────────────────────────────┤
│                   Payload Data                      │
└─────────────────────────────────────────────────────┘
```

### таблиця Opcode
| Opcode | Назва | Призначення |
|---------|---------|-------------|
| `0x1` | **Text Frame** | Передача текстових даних у форматі UTF-8 |
| `0x2` | **Binary Frame** | Передача бінарних даних (файли, зображення, аудіо тощо) |
| `0x8` | **Close Frame** | Коректне завершення WebSocket-з'єднання |
| `0x9` | **Ping Frame** | Перевірка доступності іншої сторони та підтримання з'єднання |
| `0xA` | **Pong Frame** | Відповідь на `Ping Frame` |
